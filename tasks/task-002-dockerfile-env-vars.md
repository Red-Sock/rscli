# Task 002 — Dockerfile: emit ENV directives from project config on tidy

## Goal

During `rscli project tidy`, regenerate the project's `Dockerfile` so it contains a Docker `ENV` instruction for every variable defined in `cfg.Environment`, allowing values to be overridden at `docker run` time via `-e` or a container UI.

## Context

The Dockerfile is generated by `GenerateDockerfile` in:
`plugins/project/go_project/patterns/generators/dockerfile_generator/service_dockerfile.go`

It uses `text/template` with the template at:
`plugins/project/go_project/patterns/generators/dockerfile_generator/templates/Dockerfile.pattern`

`PrepareDockerfile` (in `plugins/project/actions/go_actions/preparation.go`) is already in the tidy pipeline (`goProjectTidyActions()`), but it **skips regeneration** if the file already exists on disk:
```go
if dockerFile != nil {
    return nil
}
```

`cfg.Environment` is `matreshka.Environment` (`[]*environment.Variable`). Each `Variable` has:
- `Name string` — snake_case name (e.g. `log_format`, `log_level`)
- `Value` — has a `.String()` method returning the default value as a string

The desired Docker output (placed after `COPY` / `EXPOSE`, before `ENTRYPOINT`):
```dockerfile
ENV LOG_FORMAT=TEXT \
    LOG_LEVEL=Info
```

Variable name conversion: `strings.ToUpper` + replace spaces and dots with `_`.

Snapshot files used by the existing test live at:
`plugins/project/go_project/patterns/generators/dockerfile_generator/test_snapshots/`

## Acceptance Criteria

- [ ] `GenerateDockerfile` extracts `cfg.Environment` and populates an `EnvVars []EnvVar` (or equivalent) field on `serviceDockerfileArgs`.
- [ ] `Dockerfile.pattern` template renders a single `ENV` instruction (multi-line backslash form) when `EnvVars` is non-empty; renders nothing when `EnvVars` is empty.
- [ ] Variable names are uppercased with `_` word separator (e.g. `log_format` → `LOG_FORMAT`).
- [ ] `PrepareDockerfile.Do()` **always regenerates** the Dockerfile (remove the early-return guard), so tidy keeps it in sync with config changes.
- [ ] All existing snapshot tests in `dockerfile_generator` still pass (update snapshot files as needed to include the new `ENV` block).
- [ ] A new test case covers a project with environment variables and verifies the `ENV` block in the output.
- [ ] `go test ./plugins/project/go_project/patterns/generators/dockerfile_generator/...` passes.

## Files to Create / Modify

- `plugins/project/go_project/patterns/generators/dockerfile_generator/service_dockerfile.go` — add `extractEnvVars`, extend `serviceDockerfileArgs`
- `plugins/project/go_project/patterns/generators/dockerfile_generator/templates/Dockerfile.pattern` — add `{{ if .EnvVars }}ENV ...{{ end }}` block
- `plugins/project/go_project/patterns/generators/dockerfile_generator/test_snapshots/*.Dockerfile` — update snapshots if env vars are present in mock config
- `plugins/project/go_project/patterns/generators/dockerfile_generator/service_dockerfile_test.go` — add test case with env variables
- `plugins/project/actions/go_actions/preparation.go` — remove early-return guard in `PrepareDockerfile.Do()`

## Do NOT change

- `plugins/project/go_project/patterns/generators/dockerfile_generator/const.go`
- Any other generator outside `dockerfile_generator/`
- `support/compiler/main.go`

## Notes

- Use `strings.ToUpper(strings.ReplaceAll(v.Name, " ", "_"))` for name conversion (spaces were already normalised by matreshka's `Unmarshal`, but dots/dashes may appear).
- The `Value.String()` method on `environment.Value` returns the human-readable default (e.g. `"Text"`, `"Info"`).
- When `EnvVars` is non-empty the template should emit a single `ENV` layer (not one `ENV` per variable) to follow Docker best practices.
- Existing snapshot files that use `project_mock.GetMockProject` without env variables should not gain an `ENV` block — only add env mock options when testing the new behaviour.
